Working in a coding agent instead?

One command adds the same server to the terminal.

claude mcp add --transport http policyengine-macro https://policyengine--policyengine-macro-mcp-serve.modal.run/mcp

All thirteen tools on the server, and which models they reach
scoring

score_reform — one PolicyEngine reform dict, scored by model="obr" (macroeconometric emulator, via the static-costing bridge), "microsim" (static population costing) or "og". Every result carries the same score block, so models are comparable side by side.

obr emulator

obr_shock shocks one model variable directly (CGG, CGIPS, TCPRO); list_reform_variables returns those variables with their units. Units differ per variable — call it first.

uk svar

forecast_uk (GDP growth and CPI fan bands), latest_shocks (sign probabilities for the six identified shocks), model_summary (committed replication headlines, instant).

frb/us

frbus_shock shocks one FRB/US lever and returns US impulse responses, under policy_rule inertial_taylor (default), taylor or fixed_funds_rate — the rule materially changes the answer. frbus_list_variables returns the levers and their units; frbus_summary gives vintage and validation provenance. Raw variable shocks only: it does not score PolicyEngine reforms.

microsimulation

calculate_household, household_reform_impact, population_reform_impact, and list_reform_parameters for verified parameter paths, units and live baseline values.

Four of the five models are reachable here, and FRB/US is reachable only as raw variable shocks — score_reform refuses model="frbus" and points at frbus_shock, because no bridge from a PolicyEngine US reform to FRB/US fiscal levers exists. OG-UK is the one model excluded from hosting, because one solve takes tens of minutes — asking score_reform for model="og" returns local install instructions rather than a result.

three things that will trip you up

country is required

There is no default on score_reform, calculate_household or household_reform_impact: pass "uk" or "us". They are different tax-benefit models, so the server refuses rather than guessing. (population_reform_impact alone defaults to "uk".)

reform dict shapes

Flat — {"gov.hmrc.income_tax.rates.uk[0].rate": 0.21} — or a single effective date: {"gov.hmrc.income_tax.rates.uk[0].rate": {"2026-01-01": 0.21}}. Date ranges are rejected: reform values have no expiry, so a range would silently score a permanent change. For a time-limited reform, score each affected year separately.

it scales to zero

The first call after an idle spell spends ~10s waking the container. Then: list_*, model_summary and frbus_summary are instant; frbus_shock ~3s cold and well under a second warm; households a few seconds; forecast_uk and population_reform_impact tens of seconds; score_reform with model="obr" a minute or more. Set a generous client timeout.

or — a terminal instead

Same models, one command.

One package (Python 3.10+) wraps the OBR emulator, the UK SVAR, FRB/US and PolicyEngine behind a single command, models and data included — no cloning, except for FRB/US (see below). Claude Code and Codex can drive it from their own terminal, or you can run it yourself.

pip install "policyengine-macro[models] @ git+https://github.com/PolicyEngine/macro#subdirectory=integration"
pe-macro obr-shock --var CGG --shock 1250 --periods 4

That prints per-quarter GDP deltas in £bn and the cumulative effect — a table by default, JSON with --json. The CLI runs everything the MCP server does, plus OG-UK, which is too slow to host. Unlike the MCP tools, CLI --country defaults to uk; reform dicts take the same shapes as the MCP tools — flat or single-effective-date, never a date range.

Every command, and what installs with what

Beyond obr-shock: score (a PolicyEngine reform through --model og|obr|microsim) and compare; variables; forecast, shocks, summary; household, household-impact, population-impact, parameters; frbus-shock, frbus-variables and frbus-summary; and og-score / og-baseline, which need OG-UK installed and take tens of minutes.

The [models] extra pulls the OBR emulator, the UK SVAR and the PolicyEngine country packages, each of which ships its own data. OG-UK installs separately. The extra also pulls FRB/US, but that one must be installed editably from a checkout: its model.xml and LONGBASE.TXT live in the repository's vendor/ directory rather than inside the wheel, so a plain install imports but has no model to run. The hosted server has no such caveat. The package lives in this repo's integration/ directory; a shorter pip install policyengine-macro follows PyPI publication.

all three routes call the same code — integration/ in the repo.

or — write the code yourself

The Python API, in one screen.

No connector, no CLI — just Python. Every model is its own installable package. Pick one below: install, then the whole call in one screen, with the mechanics and options behind the expander underneath.

Install straight from GitHub, no clone needed (Python 3.11–3.13). The whole scoring loop is four moves:

01 · buildthe reforma PolicyEngine parameter + a new value
02 · solveboth worldsbaseline and reform steady states
03 · mapto £bnmodel units → current-price aggregates
04 · readthe impactlevel, change, and % per aggregate
install 
pip install git+https://github.com/PSLmodels/OG-UK
score_reform.py 
from datetime import datetime
from policyengine.core import ParameterValue, Policy
from policyengine.tax_benefit_models.uk import uk_latest
from oguk import solve_steady_state, map_to_real_world

# 01 BUILD — a reform is real statute: a PolicyEngine parameter + a new value.
#          Here: basic rate of income tax 20% → 21% from 2026.
param = uk_latest.get_parameter("gov.hmrc.income_tax.rates.uk[0].rate")
reform = Policy(name="Basic rate 21%", parameter_values=[
    ParameterValue(parameter=param, value=0.21,
                   start_date=datetime(2026, 1, 1))])

# 02 SOLVE — two general-equilibrium solves (~5-15 min each).
#          No policy argument = current law. Same call + policy = the reform.
baseline  = solve_steady_state(start_year=2026)
reform_ss = solve_steady_state(start_year=2026, policy=reform)

# 03 MAP  — model units → current-price £bn, anchored to ONS/OBR data.
impact = map_to_real_world(baseline, reform_ss)

# 04 READ — level, change, and % change for each aggregate.
print(f"GDP:     {impact.gdp_change:+.1f}bn ({impact.gdp_pct:+.3f}%)")
print(f"Revenue: {impact.tax_revenue_change:+.1f}bn")
print(f"Debt:    {impact.debt_change:+.1f}bn")
output · illustrative 
GDP:     +2.9bn (+0.041%)
Revenue: +6.4bn
Debt:    −3.1bn
How it works, the transition path, and the options

Under the hood: PolicyEngine runs the reform through the actual UK tax-benefit rules on representative microdata, OG-UK estimates tax functions from that, and the OLG model solves for how 80 age cohorts and firms re-optimise — giving you the new equilibrium wages, interest rate, and aggregates.

mode

solve_steady_state() for the long-run answer; run_transition_path() for the year-by-year fiscal profile (slower, runs on Dask).

age_specific

"pooled" (default, fastest) → "brackets" → "each" for finer age-specific tax functions.

param_overrides

Structural shocks that aren't statute: {"cit_rate": [[0.25]]} for corporation tax, {"Z": [[1.004]]} for TFP.

The OBR emulator ships its equation listing and forecast data inside the package — one pip install, no clone — then a reform is a single shock to an exogenous variable:

01 · picka leveran exogenous variable to shock
02 · shockthe valuea spend increase or a rate change
03 · solvethe systemGauss–Seidel, shocked vs baseline
04 · readthe deviation£bn impact, quarter by quarter
install 
pip install git+https://github.com/PolicyEngine/obr-macroeconomic-model
run_reform.py 
from obr_macro import run_reform

# A £5bn government spending shock: £1.25bn per quarter over 4 quarters.
# CGG is the OBR's exogenous government-consumption variable.
results = run_reform(
    name="Fiscal Stimulus",
    var="CGG",
    shock=1250,
    periods=4,
)

# Deviation from baseline, quarter by quarter — the fiscal multiplier.
print(results[["period", "delta_gdp_bn", "pct_gdp"]])

How it works, and the options

Under the hood: 372 transpiled equations, solved simultaneously by Gauss–Seidel each quarter, with the closure swapping for a shock so that GDP becomes endogenous rather than held fixed. The model page covers the solver, the anchoring and every lever.

var

The exogenous lever to shock: "CGG" (gov. consumption), "CGIPS" (public investment), "TCPRO" (corporation tax rate).

periods

How many quarters to solve — the multiplier builds over the first few quarters and settles.

investment_closure

Set True to switch on the investment channel for tax-rate shocks like a corporation-tax change.

The structural VAR doesn't score reforms — it's the suite's baseline/conditioning member: it reads the economy. One pip install — the assembled dataset and latest results ship inside the package — and the pipeline is four moves:

01 · assemblethe data8 quarterly series from ONS, BoE, FRED
02 · estimatethe BVARMinnesota/NIW prior, posterior draws
03 · identifythe shockszero + sign restrictions, six named shocks
04 · readthe economyIRFs, decompositions, fan-chart forecasts
install 
pip install git+https://github.com/PolicyEngine/boe-var-model
run the pipeline 
# the packaged dataset loads out of the box — no download step
from boe_var.data import load_data
from boe_var.bvar import BVAR
from boe_var.identification import identify

df = load_data()                    # 8 quarterly series, through 2024Q2
model = BVAR(df.to_numpy(), lags=4)
draws = model.sample_posterior(1000)
pairs = identify(draws)             # zero + sign restrictions, 6 shocks

How it works, and what ships in the package

Under the hood: an 8-variable Bayesian VAR — estimated on 1992Q1–2023Q2 with Covid dummies, leaving the 2023Q3–2024Q2 quarters of the packaged dataset for out-of-sample checking — is sampled from its normal-inverse-Wishart posterior, and each draw is rotated per Arias–Rubio-Ramírez–Waggoner (2018) until the zero and sign restrictions name six structural shocks. The PolicyEngine Macro CLI wraps the common outputs: pe-macro forecast, pe-macro shocks, pe-macro summary.

run_replication.py

The paper's Figures 2–6: IRFs, variance decompositions, estimated shocks, and historical decompositions of UK GDP and inflation.

run_forecast_revision.py

The forecasting toolkit: fan charts, P(sign) of the latest quarter's shocks, composite IRFs, and the Section-5 forecast-revision split.

boe_var

The library underneath: load_data(), the BVAR class, identify(), and the IRF/FEVD/decomposition analysis functions.

The micro member of the suite: PolicyEngine's tax-benefit microsimulation runs the actual UK and US statute for a specific family — the same engine behind policyengine.org. One pip install, and a household calculation is one call:

01 · describethe householdages, incomes, family structure
02 · calculatethe statuteevery tax and benefit, current law
03 · reforma parameterre-run the same family under new law
04 · readthe impactnet income, person by person
install 
pip install policyengine
household.py 
import policyengine as pe

# UK: a single adult earning £50,000 — current law
uk = pe.uk.calculate_household(
    people=[{"age": 35, "employment_income": 50_000}],
    year=2026,
)
print(uk.person[0].income_tax)
print(uk.household.hbai_household_net_income)

# US: same idea, with a reform applied
us = pe.us.calculate_household(
    people=[{"age": 35, "employment_income": 60_000}],
    tax_unit={"filing_status": "SINGLE"},
    household={"state_code": "CA"},
    year=2026,
    reform={"gov.irs.credits.ctc.amount.adult_dependent": 1000},
)
print(us.tax_unit.income_tax, us.household.household_net_income)
How it works, and the options

Under the hood the call resolves a dependency graph of statutory rules for the household you describe — the model page has the mechanics. Worth knowing here: the same package runs whole representative datasets, and population scoring works on the hosted server with no setup (the gated-microdata credential is provisioned server-side) but needs your own HUGGING_FACE_TOKEN locally.

calculate_household

pe.uk and pe.us — taxes, benefits, and net income for one family, with an optional reform dict.

reform

A parameter path and a new value: {"gov.hmrc.income_tax.rates.uk[0].rate": 0.21}, or a single effective date: {"...rate": {"2026-01-01": 0.21}}. Date ranges are not a shape — values never expire. Same paths the OLG model scores.

Simulation

Population analysis: run a dataset like enhanced_frs_2023_24 through the model and aggregate any variable by decile or entity.
